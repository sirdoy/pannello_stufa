<objective>
Close 2 UAT gaps from 144-UAT.md:

1. **UAT#5 (minor)**: LightsCard footer positioned mid-card ("yes, ma è posizionato a metà card e non alla fine"). Root cause: `<LastUpdated>` rendered as DeviceCard child, but DeviceCard renders `footerActions` ("Tutte le Stanze e Scene →") *after* children, so the timestamp appears above the button.

2. **UAT#8 (major)**: ThermostatCard missing temperature + footer at wrong position ("no e mancano anche dei dati nella card, come la temperatura"). Two root causes:
   - `LastUpdated` rendered as DeviceCard child but DeviceCard renders `infoBoxes` (Casa/Stanze/Dispositivi) *after* children, pushing timestamp into middle.
   - `useThermostatData` uses WS-primary + polling-fallback with `interval: isWsConnected ? null : ...`. When WS connects before initial HTTP fetch runs and the netatmo WS topic does not push an initial snapshot, `status` stays `null` forever → `selectedRoom.temperature` undefined.

Solution: add a `footerContent` slot to DeviceCard rendered AFTER infoBoxes and footerActions; migrate both cards to use it. Decouple thermostat's initial status fetch from the polling gate.

Other cards (StoveCard, NetworkCard, SonosCard, DirigeraCard) use `SmartHomeCard` directly — no DeviceCard-ordering issue, no changes needed.
</objective>

    Ensure `status` is populated at least once whether WS is connected or not. The existing polling gate suppresses polling when WS is OPEN, which means if the netatmo WS topic never broadcasts an initial snapshot the card never shows temperature data.

    **Add** an `initialStatusFetchedRef` guard and a new `useEffect` that runs `fetchStatus` exactly once after `topology` is loaded, independent of WS state.

    Insert after the existing connection-check effect (around line 153) and before `fetchStatus` declaration:

    ```typescript
    const initialStatusFetchedRef = useRef(false);

    useEffect(() => {
      if (initialStatusFetchedRef.current) return;
      if (!topology) return;
      initialStatusFetchedRef.current = true;
      fetchStatus();
    // eslint-disable-next-line react-hooks/exhaustive-deps
    }, [topology]);
    ```

    Keep the existing `useAdaptivePolling` block unchanged — it still handles the steady-state polling when WS is closed. The new effect only guarantees the first fetch.

    **Why not also fetch inside the WS subscribe effect?** The WS path already calls `setStatus` when messages arrive, but relies on the backend to broadcast. The initial HTTP fetch is a safety net that guarantees `status` is populated regardless of WS backend behavior, without creating a polling loop.
