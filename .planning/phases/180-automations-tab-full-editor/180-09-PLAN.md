<!-- Source-verified from app/components/ui/ConfirmationDialog.tsx (see 180-07-PLAN <confirmation_dialog_contract>). -->

ConfirmationDialog renders via **Radix `DialogPrimitive`** — DOM role is `dialog`, NOT `alertdialog`.

**RULE: Never use `.first()` / `.last()` to disambiguate when a label appears in both the editor footer and the dialog.**

When ConfirmationDialog opens (delete confirm or unsaved-changes), the DOM contains:
- Editor footer's "Annulla" button (always)
- Editor footer's "Elimina" button (in edit mode)
- Dialog's confirm button — label is whatever was passed in `confirmLabel` (e.g. "Elimina" for delete, "Chiudi senza salvare" for unsaved)
- Dialog's cancel button — label is whatever was passed in `cancelLabel` (e.g. "Annulla" for delete, "Continua a modificare" for unsaved)

For **unsaved-changes dialog** (default variant), the labels do NOT collide with editor labels:
- `confirmLabel: "Chiudi senza salvare"` — unique
- `cancelLabel: "Continua a modificare"` — unique
→ Direct queries are safe: `page.getByRole('button', { name: 'Continua a modificare' })`

For **delete confirm dialog** (danger variant), labels DO collide:
- `confirmLabel: "Elimina"` — collides with editor footer's "Elimina" button
- `cancelLabel: "Annulla"` — collides with editor footer's "Annulla" button
→ MUST scope to the dialog. Two acceptable patterns:

**Pattern A (preferred — scoped role query):**
```ts
const dialog = page.getByRole('dialog');
await dialog.getByRole('button', { name: 'Elimina' }).click();
// or
await dialog.getByRole('button', { name: 'Annulla' }).click();
```

**Pattern B (testid — also acceptable, especially for keyboard automation):**
```ts
await page.getByTestId('confirmation-confirm').click();
await page.getByTestId('confirmation-cancel').click();
```

**FORBIDDEN patterns (will fail on Radix Portal placement changes):**
```ts
// ❌ relies on DOM order; Radix Portal may insert at start or end of body
await page.getByRole('button', { name: 'Elimina' }).last().click();
await page.getByRole('button', { name: 'Annulla' }).first().click();
```

Heading inside dialog (visible `<h2>`):
```ts
await expect(page.getByRole('dialog').getByRole('heading', { level: 2, name: /Eliminare/ })).toBeVisible();
// or simpler text-match if heading content is unambiguous:
await expect(page.getByText(/Eliminare l'automazione "Sveglia mattutina"/)).toBeVisible();
```

When the editor itself is open inside a Sheet, the dialog appears at z-index 50 above the Sheet (200/201). Both share the DOM. Always scope ConfirmationDialog queries via `page.getByRole('dialog')` (or testids) — never via DOM-order selectors.

  id: 1,

async function mockAutomationsApi(page: Page, opts: { rules?: object[] } = {}) {
  const rules = opts.rules ?? [MOCK_RULE_BASE];
  await page.route('**/api/v1/automations**', async (route) => {
    const method = route.request().method();
    const url = new URL(route.request().url());
    if (method === 'GET' && !url.pathname.includes('/executions') && !/\/\d+$/.test(url.pathname)) {
      await route.fulfill({
        status: 200,
        contentType: 'application/json',
        body: JSON.stringify({ items: rules, total_count: rules.length, limit: 20, offset: 0 }),
      });
    } else if (method === 'POST') {
      const body = JSON.parse(route.request().postData() ?? '{}') as Record<string, unknown>;
      await route.fulfill({
        status: 201,
        contentType: 'application/json',
        body: JSON.stringify({ ...MOCK_RULE_BASE, ...body, id: 99 }),
      });
    } else if (method === 'PATCH') {
      await route.fulfill({
        status: 200,
        contentType: 'application/json',
        body: JSON.stringify({ ...MOCK_RULE_BASE }),
      });
    } else if (method === 'DELETE') {
      await route.fulfill({ status: 204, body: '' });
    } else {
      await route.continue();
    }
  });
}
