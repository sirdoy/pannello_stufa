# Home WebSocket coverage audit — Raspi polling-suppression fix

## Context

User asked: dashboard home should fetch as much as possible via WebSocket and use REST endpoints only as fallback when WS is unavailable.

Audit of all 10 home cards (`app/components/DashboardCards.tsx`):

| Card | Hook | WS topic | Polling suppressed when WS up |
|------|------|----------|-------------------------------|
| Stufa | useStoveData | thermorossi | ✅ |
| Termostato | useThermostatData | netatmo | ✅ |
| Luci | useLightsData | hue | ✅ |
| Sonos | useSonosData | sonos | ✅ |
| DIRIGERA | useDirigeraData | dirigera | ✅ |
| Rete | useNetworkData | fritzbox | ✅ |
| Tuya | useTuyaData | tuya | ✅ |
| Raspi | useRaspiData | raspi | ❌ — polls always |
| Camera | useCameraData | (no topic) | n/a (REST only) |
| Meteo | useWeatherSummary | (no topic) | n/a (external Open-Meteo) |

HA proxy WS topics per `docs/api/websocket.md`: `fritzbox, dirigera, netatmo, thermorossi, tuya, hue, sonos, raspi, scheduler, sonos_transport, sonos_volume, sonos_topology, automations`. **No `camera` topic, no `weather` topic.** The Netatmo topic carries energy/thermostat data only (rooms with `therm_*` fields) — Netatmo Welcome/Presence camera state is REST-only via `/api/v1/netatmo/camera/status`. Camera + Weather therefore stay polling-only by server constraint and are out of scope.

The single real gap is `useRaspiData`: it subscribes to the `raspi` WS topic correctly but `useAdaptivePolling` keeps polling unconditionally. The hook also carries a stale comment claiming "Raspi is not in HA proxy WS topics" — incorrect since `raspi` is a published topic.

## Tasks

### Task 1: Suppress Raspi polling when WS is OPEN

**Files:**
- `app/components/devices/raspi/hooks/useRaspiData.ts`
- `app/components/devices/raspi/hooks/__tests__/useRaspiData.test.ts` (verify)

**Action:**
1. In `useRaspiData.ts`:
   - Replace the stale comment block above `useAdaptivePolling` with the canonical "WS-primary, polling fallback" comment used by sibling hooks (e.g. `useTuyaData`, `useDirigeraData`).
   - Change `interval: interval` to `interval: isWsConnected ? null : interval` so polling is paused while the WebSocket is OPEN. `useAdaptivePolling` accepts `null` to mean "do not poll" (verified in `lib/hooks/useAdaptivePolling.ts`).
2. Inspect the existing test file. If a test asserts that polling fires regardless of WS state, update or add a test mirroring `useTuyaData.test.ts` / `useDirigeraData.test.ts` patterns: when `readyState === OPEN`, the polling callback must not fire on its interval.

**Verify:**
- `npm run test:unit -- app/components/devices/raspi/hooks/__tests__/useRaspiData.test.ts`
- `npx tsc --noEmit` (no new TS errors)
- Grep confirms no other call site of `useRaspiData` relies on the old polling-always behavior

**Done when:**
- `useRaspiData` polling is suppressed while WS is OPEN, falling back to REST polling on disconnect/CLOSED — same pattern as the other 7 device hooks
- Tests pass
- Stale comment removed

## Out of scope

- **Camera + Weather**: no server-side WS topic exists. Migrating these would require new HA proxy topics (`camera`, `weather`) and is a separate, bigger task.
- All 7 already-WS-primary hooks: no change needed.
